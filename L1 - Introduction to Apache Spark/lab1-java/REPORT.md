# Отчет по лабораторной работе 1

## Тема

Введение в Apache Spark. Анализ данных велопарковок Сан-Франциско на Java.

## Цель работы

Цель лабораторной работы - познакомиться с базовыми возможностями Apache Spark и реализовать аналитическую обработку CSV-данных о поездках на велосипедах.

В ходе работы требовалось обработать два набора данных:

- `trips.csv` - информация о поездках;
- `stations.csv` - информация о станциях велопарковок.

## Постановка задачи

По заданию необходимо было решить пять задач:

1. Найти велосипед с максимальным временем пробега.
2. Найти наибольшее геодезическое расстояние между станциями.
3. Найти путь велосипеда с максимальным временем пробега через станции.
4. Найти количество велосипедов в системе.
5. Найти пользователей, потративших на поездки более 3 часов.

Решение должно быть выполнено на Java. Для проверки корректности были добавлены тесты.

## Выбранный подход

Для решения использован Apache Spark в локальном режиме `local[*]`.

Spark не запускался как отдельный кластер. При запуске тестов или приложения создается локальный `JavaSparkContext`, который выполняет обработку в рамках одного JVM-процесса. Такой режим удобен для лабораторной работы, потому что не требует отдельной настройки Spark-кластера, но позволяет использовать те же базовые Spark API.

Spring Boot не использовался, так как задача не является веб-приложением и не требует REST API, DI-контейнера или фонового сервиса. Для одноразовой batch-обработки достаточно обычного Java-приложения с Maven.

## Описание реализации

### Чтение данных

Данные поездок и станций читаются как текстовые файлы. Заголовки CSV-файлов отбрасываются, после чего каждая строка преобразуется в Java-объект:

- строка из `trips.csv` преобразуется в `Trip`;
- строка из `stations.csv` преобразуется в `Station`.

Парсинг вынесен в отдельный класс `CsvParsers`, чтобы упростить тестирование и отделить обработку CSV от Spark-логики.

### Задача 1. Велосипед с максимальным временем пробега

Для каждой поездки берется идентификатор велосипеда и длительность поездки. Далее данные агрегируются по `bikeId`, суммируется общее время поездок, после чего выбирается велосипед с максимальной суммой.

Результат представлен моделью `BikeRideDuration`.

### Задача 2. Максимальное расстояние между станциями

Для всех пар станций вычисляется геодезическое расстояние по координатам. Для расчета используется формула haversine.

После перебора пар выбирается пара станций с максимальным расстоянием.

Результат представлен моделью `StationDistance`.

### Задача 3. Путь велосипеда с максимальным временем пробега

Сначала используется результат первой задачи: определяется велосипед с максимальным суммарным временем поездок.

Затем из общего набора поездок выбираются только поездки этого велосипеда. Они сортируются по времени старта, после чего формируется путь через станции.

Результат представлен моделью `BikePath`.

### Задача 4. Количество велосипедов

Из поездок извлекаются идентификаторы велосипедов, затем выполняется `distinct` и подсчет уникальных значений.

### Задача 5. Пользователи с суммарным временем поездок более 3 часов

В исходном наборе данных нет отдельного поля `user_id`, поэтому пользователь был представлен как комбинация:

```text
subscription_type + zip_code
```

По этой паре выполняется группировка, суммируется время поездок, затем остаются только пользователи с суммарной длительностью больше 3 часов.

Результат представлен моделью `UserDuration`.

## Проверка работы

Для проверки добавлены тесты:

- `CsvParsersTest` - проверяет корректность разбора CSV-строк;

## Ограничения и замечания

Поле пользователя в исходных данных отсутствует, поэтому для пятого задания используется приближенная идентификация пользователя по паре `subscription_type + zip_code`.

Spark запускается локально, а не в кластере. Для лабораторной работы этого достаточно, потому что проверяется корректность обработки данных и использование Spark API.
